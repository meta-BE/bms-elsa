# サーバーサイドページネーション実装の反省（2026-02-28）

## 何をしたか
- SongTable.svelteのPAGE_SIZEを5000→100に変更し、スクロール連動でページ単位取得する方式に移行
- TanStack Tableを廃止し、シンプルなColumn型配列 + 手動レンダリングに変更
- pageCacheとgenerationカウンタによるキャッシュ管理を実装

## 何が起きたか
- バックエンドからのデータ読み込みが異常に遅くなった

## 根本原因
- ListSongsのSQLクエリが重い（CTE song_groups + correlated EXISTS + COUNT(*) OVER()）
- 旧実装ではこの重いクエリが1回だけ実行されていた（PAGE_SIZE=5000で全件取得）
- 新実装ではスクロールのたびにクエリが実行され、最大27回（2666件÷100件）になった
- LIMIT/OFFSETはSQLiteでは結果の切り出しのみで、CTE/ウィンドウ関数の実行コストは削減されない

## 教訓
1. **SQLの計算コストとページサイズの関係を事前に評価すべき**: GROUP BY + ウィンドウ関数 + 相関サブクエリを含むクエリでは、LIMIT/OFFSETによるページネーションは実行回数が増えるだけで1回あたりのコストは変わらない
2. **データ量が少ない場合（~2,666件）は一括取得で十分**: ネットワーク/IPC転送量は約600KBで問題ない
3. **パフォーマンス改善では、ボトルネックの正確な特定が先**: 今回は「初回描画の高速化」が目的だったが、実際のボトルネックがSQLクエリ自体なのかフロントエンド描画なのかを計測せずに設計に進んでしまった

## 再利用可能なパターン
- TanStack Table廃止 → シンプルなColumn型配列への変更は有効。コードが簡潔になり、ビルドサイズも削減された
- animate-pulseプレースホルダーの実装パターン
- generationカウンタによるstale response破棄パターン
