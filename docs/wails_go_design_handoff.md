# Wails + Go デスクトップアプリ設計 引き継ぎ資料

## 1. プロジェクト概要

### 目的

ローカルファイルを扱う軽量デスクトップアプリの開発。\
配布サイズを最小化しつつ、高速・シンプルな設計を目指す。

### 想定機能

-   フォルダ走査
-   ファイル一覧表示（数千〜数万件想定）
-   リネーム
-   移動
-   メタ情報取得

UIはWebベース（Wails）。

------------------------------------------------------------------------

## 2. 技術選定の経緯

### JVM（Kotlin + jlink）は不採用

-   `jlink` によるランタイム削減を検討
-   依然として配布サイズが大きい
-   JVM依存が残る

→ バンドルサイズ観点で不採用

------------------------------------------------------------------------

### Electronは不採用

-   Chromium + Node 同梱
-   配布サイズが大きい（100MB級）
-   メモリ消費が重い

→ 軽量志向と合わない

------------------------------------------------------------------------

### Wails + Go 採用理由

-   WebViewベース（Chromium同梱不要）
-   バイナリサイズが小さい
-   Goのみで完結
-   Rust学習コスト不要（Tauriより低い）

------------------------------------------------------------------------

## 3. 非機能要件（重要）

-   バンドルサイズ最小化
-   高速起動
-   メモリ使用量低減
-   大量ファイル処理耐性
-   シンプルな依存構成

UI豪華さより設計品質重視。

------------------------------------------------------------------------

## 4. 想定アーキテクチャ

    /cmd
    /internal
      /app        ← Wailsバインディング層
      /service    ← ビジネスロジック
      /fs         ← ファイル操作抽象
      /model

### 原則

-   Wails層にビジネスロジックを書かない
-   IPC境界を明確化
-   ファイル操作は抽象化
-   テスト可能な構造にする

------------------------------------------------------------------------

## 5. IPC設計方針

避けるべき：

-   1ファイル1RPC呼び出し
-   巨大JSONの頻繁往復

推奨：

-   バッチAPI
-   ページング
-   差分更新
-   イベント駆動型通知

------------------------------------------------------------------------

## 6. 大量ファイル表示設計

問題点：

-   JSONシリアライズコスト
-   UI再描画コスト
-   メモリ圧迫

対策：

-   ページネーション or 仮想スクロール
-   フィルタリングはGo側で実施
-   UIには必要最小限のDTOのみ送る

------------------------------------------------------------------------

## 7. 並行処理設計（Go活用）

-   goroutineで走査並列化
-   worker pool設計
-   contextによるキャンセル制御
-   チャネルで進捗通知

Wailsイベントでフロントへpush通知。

------------------------------------------------------------------------

## 8. バイナリサイズ最適化

    go build -ldflags="-s -w"

追加対策：

-   不要依存を排除
-   CGOを避ける
-   必要ならUPX圧縮

------------------------------------------------------------------------

## 9. 次フェーズ設計タスク

1.  クリーンアーキテクチャ詳細設計
2.  大量ファイル一覧API設計
3.  イベント駆動型進捗通知設計
4.  最小バイナリ構成検証

------------------------------------------------------------------------

## まとめ

-   JVMとElectronはサイズ面で不採用
-   TauriよりWails（Go）を選択
-   最重要関心はバンドルサイズと軽量性
-   大量ファイル処理を前提とした設計が必要
-   IPC設計と責務分離が鍵
